# Dualboot

SC-OBC supports a dualboot configuration designed for high reliability
and safe system updates.

The system maintains two bootable system images:

- **golden**
- **main**

The **golden image** acts as a recovery system and is not normally updated.
The **main image** is the primary system that receives updates.

---

## Overview

The dualboot system works as follows:

1. The system normally boots from the **main** slot.
2. During updates, the system temporarily switches to the **golden** system.
3. The update bundle is downloaded and installed to the **main** slot.
4. After installation, the system automatically reboots into the updated **main** system.

This design ensures that a known-good recovery system is always available
while updating the primary system safely.

## Partition layout

A typical layout is:

```text
┌───────────────────────────────────────────────────────────────┐
│  boot_golden  (VFAT, /boot from golden rootfs)                │
│      Contains BOOT.bin, BOOT0001.bin, and boot.scr-golden.    │
│      Used as recovery boot partition for fallback startup.    │
│                                                               │
│  root_golden  (EXT4, golden full rootfs)                      │
│      Generated by sc-image-golden.bb. Minimal, stable         │
│      system used for recovery or re-flashing.                 │
│                                                               │
│  data  (EXT4, shared persistent storage)                      │
│      Shared between golden and main rootfs.                   │
│      Mounted at /srv/data.                                    │
│                                                               │
│  boot_main (VFAT, /boot from main rootfs)                     │
│      Contains boot.scr-main, kernel Image, and DTB only.      │
│      Does not include any BOOT.bin.                           │
│                                                               │
│  root_main (EXT4, main full rootfs)                           │
│      Generated by sc-image-dev.bb. Main development           │
│      and active runtime environment.                          │
└───────────────────────────────────────────────────────────────┘
```

The `golden` system is intended to remain stable and unchanged,
while the `main` system is updated during normal operation.

## Build configuration

Dualboot images and update bundles are built using dedicated
`kas` configuration files.

| Configuration                   | Description                                          |
| ------------------------------- | ---------------------------------------------------- |
| `kas/<board>-dualboot.yml`      | Builds the full dualboot WIC image and update bundle |
| `kas/<board>-update-bundle.yml` | Builds only the update bundle                        |

Example:

```bash
uv run kas build kas/scobc-v1-update-bundle.yml
```

For details about available boards and configurations, see:

- [Machine Overview][docs/machines/README.md]

## Update mechanism

Updates are delivered using **RAUC** bundles.

The update process works as follows:

1. Switch the running system to golden
2. Download the update bundle
3. Install the update to the main slot
4. Automatically reboot into the updated main system

The **golden** slot is not modified during normal updates.

Updates may include the following components:

- `rootfs` (main root filesystem)
- `bootfs` (main boot filesystem)
- `BOOT.bin` stored in `boot_golden`

## Recovery behavior

If the **main** system fails to boot correctly,
the system can fall back to the golden system.

The golden system can then be used to:

- recover the device
- reinstall a valid update bundle
- perform maintenance operations

## Installing an update bundle

Update bundles can be generated with:

```bash
uv run kas build kas/<board>-update-bundle.yml
```

On the device:

```bash
sudo boot-to-golden

rauc install bundle.raucb
```

The system will automatically reboot after the installation.

## hawkbit updater

SC-OBC also supports remote updates using Hawkbit.

To enable communication between the device and the Hawkbit server,
the following environment variables must be set during the build:

| Variable                 | Description          |
| ------------------------ | -------------------- |
| `HAWKBIT_SERVER_ADDRESS` | Hawkbit server URL   |
| `HAWKBIT_TENANT_ID`      | Hawkbit tenant ID    |
| `HAWKBIT_TARGET_NAME`    | Device identifier    |
| `HAWKBIT_AUTH_TOKEN`     | Authentication token |

When these variables are configured correctly,
the target device can communicate with the Hawkbit server
and receive update bundles automatically.
